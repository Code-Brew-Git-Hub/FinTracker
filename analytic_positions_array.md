# Аналитика модуля массива позиций

## 1. Назначение

Массив позиций предназначен для детализации одной транзакции на несколько составных элементов. Это требуется для случаев, когда одна финансовая операция включает несколько товаров или услуг, которые необходимо учитывать отдельно.

Пример: одна транзакция в магазине может содержать продукты, бытовые товары и прочие покупки. Хранение такой операции как одной суммы с одной категорией снижает точность аналитики.

## 2. Цель

Целью доработки является поддержка массива позиций внутри транзакции для:
- детализации состава транзакции;
- категоризации отдельных частей одной операции;
- более точной агрегации расходов;
- сохранения поддержки транзакций без детализации.

## 3. Область применения

Поддержка массива позиций используется в следующих сценариях:
- ручная детализация транзакции пользователем;
- хранение нескольких позиций внутри одной транзакции;
- категоризация отдельных позиций;
- агрегация расходов с учетом позиций.

## 4. Функциональные требования

Система должна обеспечивать:
- получение транзакции вместе с позициями;
- создание транзакции без позиций;
- добавление одной или нескольких позиций в транзакцию;
- редактирование позиции;
- логическое удаление позиции через soft-delete;
- ручное разбиение существующей транзакции на несколько позиций;
- хранение категории на уровне позиции;
- хранение тегов на уровне позиции;
- отображение признака наличия позиций у транзакции;
- использование позиций в аналитике при их наличии;
- использование категории транзакции в аналитике, если позиции отсутствуют;
- отображение суммы транзакции и суммы позиций для пользователя.

## 5. Границы решения

### Входит в область решения
- хранение массива позиций внутри транзакции;
- CRUD-операции для позиций;
- связь позиции с транзакцией;
- категоризация позиций;
- поддержка тегов у позиций;
- передача данных в модуль агрегации.

### Не входит в область решения
- OCR и распознавание позиций из изображений и чеков;
- автоматическое извлечение позиций из PDF;
- построение графиков;
- UI-логика отображения;
- банковские интеграции.

## 6. Доменная модель

Базовая модель транзакции описана в общей аналитике проекта. В рамках данного документа фиксируются только изменения, связанные с поддержкой массива позиций.

### 6.1 Изменения на уровне Transaction

Для транзакции рекомендуется поддержать дополнительный признак:
- `has_positions` — признак наличия у транзакции массива позиций.

### 6.2 TransactionItem

`TransactionItem` — составная позиция транзакции.

Предлагаемые поля:
- `id` — идентификатор позиции;
- `transaction_id` — идентификатор транзакции, к которой относится позиция;
- `name` — наименование позиции;
- `amount` — сумма по позиции;
- `quantity` — количество единиц позиции, если применимо;
- `unit_price` — цена за единицу, если применимо;
- `category_id` — идентификатор категории позиции;
- `comment` — дополнительное текстовое описание позиции;
- `tags` — набор тегов позиции для дополнительной классификации;
- `is_deleted` — признак логического удаления позиции.

### 6.3 Связь сущностей
- одна транзакция может содержать несколько позиций;
- одна позиция принадлежит только одной транзакции;
- транзакция может существовать без позиций;
- массив позиций не является обязательным для всех транзакций.

## 7. Основные правила

### 7.1 Общие правила
- транзакция остаётся основной сущностью учёта;
- массив позиций является дочерней структурой транзакции;
- позиции используются только для детализированных транзакций;
- `scope` поддерживается только на уровне транзакции.

### 7.2 Категоризация
- если у транзакции нет позиций, аналитика по категориям строится по категории транзакции;
- если у транзакции есть позиции, аналитика по категориям строится по категориям позиций;
- если у позиции категория не задана, допускается наследование категории транзакции.

### 7.3 Теги
- транзакция может иметь собственные теги;
- позиция также может иметь собственные теги;
- теги позиции используются для дополнительной фильтрации и классификации.

### 7.4 Удаление
- удаление позиции должно быть реализовано через soft-delete;
- при логическом удалении транзакции связанные позиции также считаются удалёнными.

### 7.5 Суммы транзакции и позиций
- сумма транзакции и сумма позиций отображаются пользователю отдельно;
- система может показывать расхождение между суммой транзакции и суммой позиций;
- жёсткая блокирующая проверка совпадения сумм на текущем этапе не требуется.

## 8. Влияние на агрегацию

Модуль агрегации должен учитывать транзакции и, при наличии позиций, использовать данные позиций для более детальной аналитики.

Правила агрегации:
- если транзакция не содержит позиций, агрегация выполняется по транзакции;
- если транзакция содержит позиции, агрегация по категориям выполняется по позициям;
- наличие позиций определяется по признаку `has_positions` и связанным позициям транзакции;
- позиции, помеченные как удалённые, не участвуют в агрегации.

Такой подход позволяет учитывать состав смешанных покупок и повышает точность аналитики.

## 9. Влияние на смежные сущности

### 9.1 Scope
`scope` сохраняется на уровне транзакции и не поддерживается на уровне позиции.

### 9.2 Переводы и компенсации
Для переводов между счетами и компенсирующих транзакций массив позиций на текущем этапе не требуется.

## 10. Требования к хранению

Хранилище должно поддерживать:
- таблицу транзакций;
- таблицу позиций транзакций;
- связь one-to-many между транзакцией и позициями.

## 11. Нефункциональные требования

Решение должно быть:
- расширяемым;
- независимым от UI;
- пригодным для последующего развития аналитики.

## 13. Итоговое решение для текущего этапа

Для текущего этапа принимается следующая модель:
- транзакция остаётся основной сущностью;
- у транзакции может быть массив позиций;
- массив позиций не является обязательным;
- категория может задаваться и у транзакции, и у позиции;
- теги поддерживаются и у транзакции, и у позиции;
- `scope` остаётся только на уровне транзакции;
- при наличии позиций агрегация по категориям выполняется по позициям;
- при отсутствии позиций агрегация выполняется по транзакции.
