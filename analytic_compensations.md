# Аналитика механизма компенсирующих транзакций

## 1. Назначение

Механизм компенсирующих транзакций предназначен для связывания нескольких операций, которые в совокупности отражают один финансовый смысл и могут искажать аналитику при раздельном учете.

Примеры:
- пользователь дал в долг 1000 ₽, затем получил два возврата по 500 ₽;
- пользователь внес залог 1000 ₽, затем получил возврат 900 ₽, при этом фактический расход составил 100 ₽.

## 2. Цель

Цель механизма — дать пользователю возможность:
- связывать транзакции, которые компенсируют друг друга;
- учитывать частичную компенсацию;
- исключать компенсирующие транзакции из общей аналитики;
- получать более корректную картину доходов и расходов.

## 3. Область применения

Механизм используется в следующих сценариях:
- выдача денег в долг;
- возврат долга частями;
- внесение и возврат залога;
- иные случаи, когда одна или несколько транзакций компенсируют другую транзакцию полностью или частично.

## 4. Функциональные требования

Система должна обеспечивать:
- создание связи компенсации между транзакциями;
- возможность связывать одну транзакцию с несколькими компенсирующими транзакциями;
- возможность связывать несколько транзакций с одной итоговой транзакцией;
- поддержку частичной компенсации;
- отображение факта компенсации у транзакции;
- расчет степени компенсации по связанной группе транзакций;
- возможность исключать компенсирующие транзакции из агрегации;
- возможность учитывать компенсирующие транзакции в агрегации;
- просмотр связанных транзакций внутри одной компенсационной группы;
- редактирование и удаление связи компенсации.

## 5. Границы решения

### Входит в область решения
- создание и хранение связей между компенсирующими транзакциями;
- поддержка частичной и полной компенсации;
- передача признака компенсации в модуль агрегации;
- управление участием компенсаций в аналитике.

### Не входит в область решения
- автоматическое определение компенсации без подтверждения пользователя;
- построение графиков;
- UI-логика отображения;
- банковские интеграции.

## 6. Доменная модель

### 6.1 CompensationGroup
`CompensationGroup` — группа связанных транзакций, которые компенсируют друг друга полностью или частично.

Предлагаемые поля:
- `id` — идентификатор группы компенсации;
- `title` — пользовательское название группы, если требуется;
- `comment` — пояснение пользователя;
- `is_closed` — признак того, что компенсация завершена;
- `created_at` — дата создания группы.

### 6.2 CompensationLink
`CompensationLink` — связь транзакции с группой компенсации.

Предлагаемые поля:
- `id` — идентификатор связи;
- `group_id` — идентификатор группы компенсации;
- `transaction_id` — идентификатор транзакции;
- `role` — роль транзакции в группе;
- `amount` — сумма транзакции, участвующая в компенсации;
- `comment` — дополнительное пояснение.

### 6.3 Роли транзакций
На текущем этапе рекомендуется поддерживать две роли:
- `base` — исходная транзакция, которую компенсируют;
- `compensation` — транзакция, компенсирующая исходную.

Такой подход достаточен для учета долгов, возвратов и залогов.

## 7. Основные правила

### 7.1 Общие правила
- одна компенсационная группа может содержать более двух транзакций;
- транзакция может участвовать только в одной компенсационной группе одного типа;
- компенсация может быть полной или частичной;
- пользователь должен видеть текущее состояние компенсации.

### 7.2 Частичная компенсация
- сумма компенсации может быть меньше суммы исходной транзакции;
- одна исходная транзакция может компенсироваться несколькими транзакциями;
- группа считается полностью компенсированной, когда сумма компенсирующих транзакций покрывает сумму исходной транзакции.

### 7.3 Залоги
- залог рассматривается как частный случай компенсирующей группы;
- если часть суммы не возвращена, невозвращенная часть считается фактическим расходом.

### 7.4 Удаление
- удаление связи компенсации должно быть логическим;
- при удалении транзакции ее связи компенсации должны считаться неактуальными;
- удаление одной связи не должно удалять остальные связи группы.

## 8. Влияние на агрегацию

Модуль агрегации должен получать информацию о том, участвует ли транзакция в компенсационной группе.

Поддерживаются два варианта расчета:
- учитывать компенсирующие транзакции в общей аналитике;
- исключать компенсирующие транзакции из аналитики.

При исключении компенсаций:
- исходные и компенсирующие транзакции, входящие в группу, не должны искажать общую картину расходов и доходов;
- для частично компенсированных случаев допускается отображение только некомпенсированного остатка, если такая логика будет поддержана позже.

На текущем этапе достаточно поддержать включение или полное исключение компенсирующих транзакций по параметру агрегации.

## 9. Влияние на смежные сущности

### 9.1 Transaction
Для транзакции должен быть доступен признак участия в компенсационной группе.

### 9.2 Scope
Компенсирующие транзакции могут одновременно принадлежать scope, но связь компенсации не заменяет scope.

### 9.3 Теги
Теги и компенсационные связи являются независимыми механизмами и могут использоваться одновременно.

## 10. Нефункциональные требования

Решение должно быть:
- расширяемым;
- независимым от UI;
- пригодным для дальнейшего развития правил компенсации.

## 11. Открытые вопросы

Требует уточнения следующий вопрос:
1. должна ли система в будущем поддерживать расчет остатка по компенсационной группе как отдельное аналитическое значение.

## 12. Итоговое решение для текущего этапа

Для текущего этапа принимается следующая модель:
- компенсации оформляются как отдельная группа связанных транзакций;
- группа может содержать более двух транзакций;
- поддерживается частичная компенсация;
- участие компенсаций в аналитике определяется параметрами агрегации;
- автоматическое определение компенсаций без подтверждения пользователя не требуется.
