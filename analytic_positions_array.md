# Аналитика модуля массива позиций

## 1. Назначение

Массив позиций предназначен для детализации одной транзакции на несколько составных элементов. Это необходимо для случаев, когда одна финансовая операция включает несколько товаров или услуг с разными категориями.

Пример: одна транзакция в магазине может содержать продукты, бытовые товары и прочие покупки. Хранение такой операции как одной суммы с одной категорией приводит к потере точности аналитики.

## 2. Цель

Целью доработки является расширение модели транзакции поддержкой вложенных позиций для:
- детализации состава транзакции;
- категоризации отдельных частей одной операции;
- более точной агрегации и аналитики расходов;
- сохранения совместимости с транзакциями без детализации.

## 3. Область применения

Поддержка массива позиций используется в следующих сценариях:
- ручная детализация транзакции пользователем;
- хранение нескольких позиций внутри одной транзакции;
- категоризация на уровне отдельных позиций;
- агрегация расходов по позициям.

## 4. Границы решения

### Входит в область решения
- хранение массива позиций внутри транзакции;
- CRUD-операции для позиций;
- связь позиции с транзакцией;
- категоризация позиций;
- проверка согласованности суммы транзакции и суммы позиций;
- передача данных в модуль агрегации.

### Не входит в область решения
- OCR и распознавание позиций из изображений и чеков;
- автоматическое извлечение позиций из PDF;
- построение графиков;
- UI-логика отображения;
- банковские интеграции.

## 5. Доменная модель

### 5.1 Transaction
Основная финансовая операция.

Предлагаемые поля:
- `id`;
- `account_id`;
- `date`;
- `amount`;
- `currency`;
- `description`;
- `transaction_type`;
- `category_id`;
- `scope_id`;
- `is_deleted`.

### 5.2 TransactionItem
Составная позиция транзакции.

Предлагаемые поля:
- `id`;
- `transaction_id`;
- `name`;
- `amount`;
- `quantity` (опционально);
- `unit_price` (опционально);
- `category_id`;
- `comment`;
- `sort_order`;
- `is_deleted`.

### 5.3 Связь сущностей
- одна транзакция может содержать несколько позиций;
- одна позиция принадлежит только одной транзакции;
- транзакция может существовать без позиций.

## 6. Основные правила

### 6.1 Общие правила
- транзакция остаётся основной сущностью учёта;
- массив позиций является дочерней структурой транзакции;
- позиции используются только для детализированных транзакций.

### 6.2 Согласованность сумм
- сумма активных позиций должна соответствовать сумме транзакции;
- временное рассогласование допустимо только в процессе редактирования;
- сохранённое корректное состояние должно обеспечивать совпадение сумм.

### 6.3 Категоризация
- если у транзакции нет позиций, аналитика строится по категории транзакции;
- если у транзакции есть позиции, аналитика по категориям строится по категориям позиций;
- если у позиции категория не задана, допускается наследование категории транзакции.

### 6.4 Удаление
- удаление позиции должно быть реализовано через soft-delete;
- при логическом удалении транзакции связанные позиции также считаются удалёнными.

## 7. Функциональные требования

Система должна обеспечивать:
- получение транзакции вместе с позициями;
- добавление позиции в транзакцию;
- редактирование позиции;
- удаление позиции через soft-delete;
- просмотр суммы транзакции и суммы позиций;
- проверку согласованности суммы транзакции и суммы позиций;
- ручное разбиение существующей транзакции на несколько позиций.

## 8. Влияние на агрегацию

Модуль агрегации должен поддерживать два режима обработки:

### 8.1 Агрегация по транзакциям
Применяется, если транзакция не содержит позиций.

### 8.2 Агрегация по позициям
Применяется, если транзакция содержит активные позиции.

Рекомендуемое правило:
- при наличии позиций агрегация по категориям выполняется по позициям;
- при отсутствии позиций агрегация выполняется по самой транзакции.

Это позволяет учитывать состав смешанных покупок и повышает точность аналитики.

## 9. Влияние на смежные сущности

### 9.1 Scope
`scope` рекомендуется сохранять на уровне транзакции, так как он описывает общее событие или группу расходов.

### 9.2 Теги
На текущем этапе теги рекомендуется оставлять на уровне транзакции.

### 9.3 Переводы и компенсации
Для переводов между счетами и компенсирующих транзакций массив позиций на текущем этапе не требуется.

## 10. Требования к хранению

Хранилище должно поддерживать:
- таблицу транзакций;
- таблицу позиций транзакций;
- связь one-to-many между транзакцией и позициями.

### Таблица `transactions`
Основные поля:
- `id`;
- `date`;
- `amount`;
- `description`;
- `transaction_type`;
- `category_id`;
- `scope_id`;
- `is_deleted`.

### Таблица `transaction_items`
Основные поля:
- `id`;
- `transaction_id`;
- `name`;
- `amount`;
- `category_id`;
- `comment`;
- `sort_order`;
- `is_deleted`.

## 11. Требования к миграции

Для поддержки массива позиций требуется миграция схемы данных.

Необходимо:
1. добавить сущность `TransactionItem`;
2. создать таблицу `transaction_items`;
3. реализовать связь транзакции с позициями;
4. сохранить совместимость со старыми транзакциями без позиций.

Рекомендуемый подход:
- не изменять исторические транзакции автоматически;
- считать, что при отсутствии позиций аналитика выполняется по транзакции;
- использовать новую таблицу только для детализированных транзакций.

## 12. Нефункциональные требования

Решение должно быть:
- совместимым с существующей моделью транзакций;
- расширяемым;
- независимым от UI;
- пригодным для последующего развития модуля аналитики.

Также необходимо:
- не нарушить обратную совместимость API;
- не ухудшить производительность агрегации;
- предусмотреть индексацию по `transaction_id`, `category_id`, `is_deleted`.

## 13. Открытые вопросы

Требуют уточнения следующие вопросы:
1. массив позиций обязателен или используется только для отдельных транзакций;
2. категория должна храниться и у транзакции, и у позиции, либо только у позиции;
3. требуется ли поддержка тегов у позиций;
4. требуется ли поддержка `scope` у позиций;
5. необходимо ли автоматически создавать одну позицию по умолчанию;
6. должны ли суммы транзакции и позиций совпадать всегда или только после завершения редактирования.

## 14. Итоговое решение для текущего этапа

Для текущего этапа рекомендуется следующая модель:
- транзакция остаётся основной сущностью;
- у транзакции может быть массив позиций;
- категория может задаваться и у транзакции, и у позиции;
- при наличии позиций агрегация выполняется по позициям;
- `scope`, теги, переводы и компенсации остаются на уровне транзакции;
- миграция выполняется через добавление новой таблицы без переработки исторических данных.

Данный подход обеспечивает более точную аналитику, не нарушая текущую архитектуру и совместимость существующих данных.
