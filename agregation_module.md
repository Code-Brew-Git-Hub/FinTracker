# Aggregation Module

## 1. Описание

Модуль агрегации отвечает за обработку и анализ транзакций.

Основная задача модуля:
- принимать набор транзакций
- применять фильтры
- агрегировать данные
- возвращать результат в удобном для клиента виде

Модуль используется для построения аналитики и подготовки данных для визуализации (графиков).

---

## 2. Основные функции

Модуль должен уметь:

- агрегировать транзакции по различным параметрам
- применять фильтрацию
- учитывать бизнес-логику (исключения)
- подготавливать данные для API

---

## 3. Входные данные

Модуль получает:

- список транзакций (из БД)
- параметры фильтрации
- параметры агрегации

Пример входных параметров:

- период (date_from, date_to)
- категории
- теги
- тип транзакции
- scope
- флаги исключения:
  - exclude_scopes
  - exclude_transfers
  - exclude_compensations

---

## 4. Фильтрация

Перед агрегацией данные должны быть отфильтрованы.

Поддерживаемые фильтры:

- по дате
- по сумме
- по категории
- по типу транзакции
- по тегам
- по scope

Дополнительно:

- исключение транзакций:
  - относящихся к определенным scope
  - являющихся переводами
  - являющихся компенсирующими (долги)

---

## 5. Агрегация

Модуль должен поддерживать несколько типов агрегации.

### 5.1 По времени

Группировка:
- по дням
- по неделям
- по месяцам

Результат:
- сумма транзакций в каждом интервале

---

### 5.2 По категориям

- группировка по category
- подсчет сумм по каждой категории

---

### 5.3 По тегам

- группировка по tag
- подсчет сумм

---

## 6. Учет бизнес-логики

Перед агрегацией необходимо учитывать:

### 6.1 Scope

Если включен флаг исключения:
- транзакции из scope не участвуют в расчетах

---

### 6.2 Переводы

- не считаются расходами или доходами
- могут полностью исключаться

---

### 6.3 Компенсации (долги)

- могут исключаться из аналитики
- важно учитывать, что одна операция может быть связана с несколькими

---

## 7. Выходные данные

Результат должен возвращаться в формате, удобном для UI.

Формат:
- JSON

Пример:

{
  "group_by": "month",
  "data": [
    { "period": "2024-01", "amount": 50000 },
    { "period": "2024-02", "amount": 42000 }
  ]
}

---

## 8. Примеры сценариев

### Сценарий 1: Траты по месяцам

Пользователь:
- задает период
- выбирает группировку по месяцам

Система:
- фильтрует транзакции
- агрегирует по месяцам
- возвращает результат

---

### Сценарий 2: Траты по категориям

Пользователь:
- выбирает категории
- включает исключение переводов

Система:
- фильтрует
- агрегирует по категориям

---

### Сценарий 3: Аналитика с учетом бизнес-логики

Пользователь:
- исключает:
  - долги
  - переводы
  - определенные группы

Система:
- убирает соответствующие транзакции
- считает только "чистые" траты

---

## 9. Требования к реализации

- модуль должен быть независимым от UI
- должен работать через сервисный слой
- должен принимать параметры агрегации через API
- должен быть расширяемым (новые типы агрегаций)

---

## 10. Производительность

Важно:

- поддержка индексов:
  - date
  - category
  - tags

- минимизация загрузки лишних данных
- по возможности использовать агрегации на уровне БД

---

## 11. Итог

Модуль агрегации — это центральная часть аналитики.

Он отвечает за:
- фильтрацию данных
- применение бизнес-правил
- построение агрегированной информации

Результат работы модуля используется для:
- отчетов
- графиков
- анализа финансов пользователя
