# Аналитика модуля агрегации

## 1. Назначение

Модуль агрегации предназначен для расчета сводных показателей по финансовым операциям пользователя.

Модуль используется для:
- расчета сумм доходов и расходов;
- группировки операций по различным признакам;
- формирования данных для аналитики и визуализации.

## 2. Цель

Цель модуля — предоставить единый механизм агрегации транзакций с учетом фильтрации и бизнес-правил.

## 3. Область решения

### Входит в область решения
- фильтрация транзакций перед расчетом;
- расчет общей суммы доходов;
- расчет общей суммы расходов;
- расчет разницы между доходами и расходами;
- агрегация по времени;
- агрегация по категориям;
- агрегация по тегам;
- агрегация по scope;
- исключение отдельных типов операций из расчета по параметрам агрегации.

### Не входит в область решения
- построение графиков;
- UI-логика отображения результатов;
- описание API-контрактов;
- вопросы производительности и технической оптимизации реализации.

## 4. Входные данные

Модуль агрегации получает:
- набор транзакций;
- параметры фильтрации;
- параметры агрегации;
- параметры исключения операций из расчета.

Поддерживаемые параметры фильтрации:
- период;
- сумма;
- категория;
- тип транзакции;
- теги;
- scope.

Поддерживаемые параметры исключения:
- исключение переводов;
- исключение компенсирующих транзакций;
- исключение транзакций из выбранных scope.

## 5. Функциональные требования

Модуль должен обеспечивать:
- фильтрацию транзакций по заданным параметрам;
- расчет общей суммы доходов;
- расчет общей суммы расходов;
- расчет разницы между доходами и расходами;
- агрегацию транзакций по выбранному признаку;
- возможность исключать переводы из расчетов;
- возможность исключать компенсирующие транзакции из расчетов;
- возможность исключать транзакции из выбранных scope;
- передачу результата в виде, пригодном для дальнейшего использования в аналитике и визуализации.

## 6. Виды агрегации

### 6.1 Агрегация по времени
Поддерживается группировка:
- по дням;
- по неделям;
- по месяцам.

Результатом является сумма операций в каждом временном интервале.

### 6.2 Агрегация по категориям
Выполняется группировка транзакций по категориям с расчетом суммы по каждой категории.

### 6.3 Агрегация по тегам
Выполняется группировка транзакций по тегам с расчетом суммы по каждому тегу.

### 6.4 Агрегация по scope
Выполняется группировка транзакций по scope с расчетом суммы по каждому scope.

Данный тип агрегации позволяет отвечать на вопросы вида:
- сколько было потрачено на конкретное событие;
- сколько средств связано с определенной поездкой, праздником или проектом.

## 7. Расчет доходов и расходов

Модуль должен поддерживать расчет следующих показателей:
- общая сумма доходов;
- общая сумма расходов;
- разница между доходами и расходами.

Под доходами понимаются транзакции, увеличивающие доступный объем средств пользователя.  
Под расходами понимаются транзакции, уменьшающие доступный объем средств пользователя.

Разница между доходами и расходами рассчитывается как:
- доходы минус расходы.

## 8. Учет бизнес-правил

### 8.1 Scope
Транзакции могут участвовать в агрегации по scope и одновременно могут быть исключены из расчета, если пользователь передал список scope, которые не нужно учитывать.

Исключение должно работать на уровне параметров агрегации:
- если scope включен в список исключаемых, связанные с ним транзакции не участвуют в расчете;
- если scope не исключен, связанные с ним транзакции участвуют в расчете на общих основаниях.

### 8.2 Переводы
Переводы между собственными счетами не должны искажать аналитику доходов и расходов, так как не отражают фактический приход или расход средств пользователя, а только перемещение денег между его счетами.

Поэтому модуль должен поддерживать возможность исключать такие транзакции из расчета.

### 8.3 Компенсации
Компенсирующие транзакции могут по-разному влиять на итоговую аналитику, поэтому модуль должен поддерживать два варианта работы:
- учитывать компенсирующие транзакции в расчетах;
- исключать компенсирующие транзакции из расчетов.

Выбор варианта должен определяться параметрами агрегации.

## 9. Сценарии использования

### 9.1 Аналитика расходов по месяцам
Пользователь задает период и выбирает группировку по месяцам.  
Модуль фильтрует транзакции, применяет правила исключения и возвращает сумму расходов по каждому месяцу.

### 9.2 Аналитика по категориям
Пользователь выбирает интересующий период и получает распределение расходов по категориям.

### 9.3 Аналитика по scope
Пользователь выбирает агрегацию по scope и получает сумму расходов по каждому событию или группе операций.

### 9.4 Аналитика без переводов и компенсаций
Пользователь включает исключение переводов и компенсирующих транзакций.  
Модуль рассчитывает более чистую картину доходов и расходов без операций, искажающих итоговую статистику.

## 10. Требования к результату агрегации

Результат работы модуля должен:
- содержать агрегированные значения по выбранному признаку;
- учитывать примененные фильтры;
- учитывать включенные исключения;
- быть пригодным для последующего использования в отчетах и визуализации.

## 11. Открытые вопросы

Требует уточнения следующий вопрос:
1. должно ли исключение компенсирующих транзакций быть включено по умолчанию, либо пользователь должен явно управлять этим параметром.

## 12. Итог

Модуль агрегации предназначен для расчета сводных финансовых показателей по транзакциям пользователя. Он должен поддерживать фильтрацию, расчет доходов и расходов, агрегацию по времени, категориям, тегам и scope, а также учитывать бизнес-правила для переводов и компенсирующих транзакций.
