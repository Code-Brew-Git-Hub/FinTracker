# Аналитика механизма проверки транзакций на корректность

## 1. Назначение

Механизм проверки транзакций на корректность предназначен для поиска операций, которые могут находиться в некорректном или сомнительном состоянии и ухудшать качество аналитики.

Пример: пользователь загружает два файла с пересекающимися периодами, из-за чего часть транзакций дублируется.

## 2. Цель

Цель механизма — дать системе возможность:
- находить подозрительные транзакции;
- предлагать пользователю проверить такие транзакции;
- не удалять данные автоматически без подтверждения;
- поддерживать гибкие правила проверки.

## 3. Область применения

Механизм используется в следующих сценариях:
- поиск полностью идентичных транзакций;
- поиск дублей после повторного импорта;
- поиск транзакций из пересекающихся периодов;
- поиск подозрительно похожих транзакций по настраиваемым правилам.

## 4. Функциональные требования

Система должна обеспечивать:
- запуск проверки транзакций на корректность;
- поиск полностью идентичных транзакций;
- поиск потенциальных дублей по набору правил;
- отображение пользователю найденных подозрительных транзакций;
- подтверждение или отклонение предложенного исправления пользователем;
- логическое удаление выбранных дублей;
- поддержку настраиваемых правил проверки;
- возможность повторного запуска проверки после импорта;
- хранение результата проверки и статуса решения пользователя.

## 5. Границы решения

### Входит в область решения
- поиск подозрительных транзакций;
- проверка по набору правил;
- отображение результата проверки;
- подтверждение действий пользователем;
- передача результата в модуль управления транзакциями.

### Не входит в область решения
- автоматическое удаление транзакций без подтверждения пользователя;
- построение графиков;
- UI-логика отображения;
- банковские интеграции.

## 6. Доменная модель

### 6.1 ValidationRule
`ValidationRule` — правило проверки транзакций.

Предлагаемые поля:
- `id` — идентификатор правила;
- `name` — название правила;
- `description` — описание правила;
- `is_enabled` — признак активности правила.

### 6.2 ValidationIssue
`ValidationIssue` — найденная потенциальная проблема.

Предлагаемые поля:
- `id` — идентификатор найденной проблемы;
- `rule_id` — правило, по которому найдена проблема;
- `transaction_ids` — список транзакций, участвующих в проблеме;
- `status` — статус обработки;
- `comment` — комментарий пользователя или системы.

### 6.3 Статусы проблемы
На текущем этапе рекомендуется поддерживать следующие статусы:
- `new` — проблема найдена и еще не обработана;
- `confirmed` — пользователь подтвердил, что требуется исправление;
- `rejected` — пользователь отклонил предложенное исправление;
- `resolved` — проблема обработана.

## 7. Основные правила

### 7.1 Общие правила
- система не должна автоматически удалять транзакции без подтверждения пользователя;
- одна и та же транзакция может участвовать в нескольких найденных проблемах;
- правила проверки должны быть отключаемыми и расширяемыми.

### 7.2 Базовые правила проверки
На текущем этапе рекомендуется поддержать как минимум:
- поиск полностью идентичных транзакций;
- поиск транзакций с одинаковой датой, суммой и описанием;
- поиск дублей после импорта файлов с пересекающимися периодами.

### 7.3 Работа пользователя с результатом
Пользователь должен иметь возможность:
- просмотреть найденные проблемы;
- согласиться с предложением удалить или исправить транзакцию;
- отказаться от предложенного исправления;
- повторно запустить проверку после новых импортов.

### 7.4 Удаление
- удаление транзакции по результату проверки должно быть логическим;
- решение пользователя по проблеме должно сохраняться;
- повторная проверка не должна без необходимости заново навязывать уже отклоненные совпадения.

## 8. Влияние на агрегацию

Модуль агрегации должен работать только с актуальными транзакциями.

Следствия:
- транзакции, удаленные пользователем по результатам проверки, не должны участвовать в агрегации;
- найденные, но не подтвержденные проблемы сами по себе не изменяют аналитику;
- до подтверждения пользователя подозрительные транзакции продолжают считаться обычными транзакциями.

## 9. Влияние на смежные сущности

### 9.1 Import
Механизм проверки должен уметь запускаться после завершения импорта.

### 9.2 Transaction
Проверка корректности не изменяет транзакцию автоматически, а только формирует предложения для пользователя.

### 9.3 Нормализация
Нормализация и проверка корректности являются разными механизмами:
- нормализация улучшает качество описаний и структуры данных;
- проверка корректности ищет потенциальные ошибки и дубли.
