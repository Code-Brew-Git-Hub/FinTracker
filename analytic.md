# 📊 Analytics — Financial Transactions Service

## 1. Введение

В рамках проекта разрабатывается self-hosted API сервис для сбора, хранения и анализа финансовых транзакций пользователя.

Сервис не привязан к конкретному интерфейсу и должен поддерживать интеграцию с различными клиентами (web, mobile, Telegram и т.д.).

Основная задача — дать пользователю инструмент для:
- загрузки транзакций из внешних источников (например, банков)
- ручного ведения учета
- структурирования данных
- получения аналитики по расходам и доходам

---

## 2. Основной пользовательский сценарий

1. Пользователь выгружает CSV файл из банка  
2. Загружает файл в сервис  
3. Сервис парсит данные и сохраняет их  
4. Пользователь:
   - дополняет недостающие транзакции
   - редактирует категории и описания  
5. Пользователь получает аналитику

---

## 3. Область применения и ограничения

### Область применения
- личные финансы
- учет расходов и доходов
- анализ трат

### Ограничения
- нет прямой интеграции с банками
- сервис работает через API
- UI не является частью MVP

---

## 4. Функциональные требования

### 4.1 Импорт данных

Система должна:
- поддерживать CSV (обязательно)
- быть расширяемой под Excel и PDF

Требования:
- разные форматы колонок
- разные разделители
- возможность маппинга колонок

Процесс:
1. загрузка файла  
2. парсинг  
3. нормализация  
4. сохранение  

---

### 4.2 Ручной ввод

- добавление одной транзакции
- дополнение данных после загрузки

---

### 4.3 CRUD операции

- Create — создание
- Read — получение
- Update — редактирование
- Delete — soft delete

---

### 4.4 Фильтрация и поиск

Фильтры:
- дата
- сумма
- категория
- тип транзакции (expense / income / transfer)
- теги
- группы (scope)

Поиск:
- по описанию
- по ключевым словам

---

### 4.5 Категории

- категории из CSV
- пользовательские категории
- редактирование категорий

Дополнительно:
- поддержка иерархии (parent-child)

---

### 4.6 Теги

- несколько тегов на транзакцию
- фильтрация по тегам

---

### 4.7 Группировка (Scope)

Пользователь может:
- объединять транзакции в группы

Примеры:
- отпуск
- ремонт
- праздник

Система должна:
- фильтровать по scope
- исключать scope из аналитики

---

### 4.8 Связи между транзакциями

#### Долги (compensations)

Пример:
- дал 1000
- вернули 500 + 500

Требования:
- связывать транзакции
- поддерживать частичные возвраты
- исключать из аналитики

---

#### Переводы

Пример:
- списание с одного счета
- зачисление на другой

Требования:
- связывать транзакции
- не учитывать как доход/расход

---

### 4.9 Массовое редактирование

Под массовым редактированием подразумевается возможность изменения сразу нескольких транзакций одним действием.

Пользователь может:
- выбрать набор транзакций
- применить к ним одно изменение

Поддерживаемые операции:
- изменение категории
- добавление/замена тегов
- привязка к группе (scope)
- добавление/изменение комментария

Важно:
- изменения применяются ко всем выбранным транзакциям одновременно
- пользователь задает одно значение, которое применяется ко всем

---

### 4.10 Нормализация

Нормализация — это процесс приведения транзакций к единому виду, чтобы убрать дубли и неконсистентные записи.

Система должна поддерживать:

1. Объединение похожих записей  
   - выявление транзакций с похожим описанием  
   - возможность привести их к одному виду  

   Пример:  
   - "McDonalds"  
   - "Mc Donalds"  
   - "McDonalds #123"  

2. Редактирование описаний  
   - изменение описания вручную  
   - возможность массового обновления описаний  

Цель:
- повысить качество данных
- улучшить точность аналитики

---

### 4.11 Аналитика

Агрегации:
- по времени
- по категориям
- по тегам

Фильтры:
- исключение scope
- исключение долгов
- исключение переводов

---

### 4.12 Визуализация

Сервис должен предоставлять данные для построения графиков.

Формат:
- данные должны возвращаться через API в агрегированном виде (JSON)
- структура должна быть удобна для UI (например: массив точек/значений)

Примечание:
- реализация графиков не является частью текущего этапа
- можно детализировать позже

---

## 5. Доменная модель (обзор)

### Transaction
- amount — сумма транзакции  
- currency — валюта транзакции (например RUB, USD)  
- date — дата и время  
- description — описание  
- category — категория  
- type — тип (expense / income / transfer)  
- source — источник (csv / manual)  
- comment — комментарий пользователя  
- is_deleted — признак soft delete  

Связи:
- tags
- scope
- links

---

### Category
- name — название категории  
- parent_id — родительская категория  

---

### Tag
- name — название тега  

---

### Scope
- name — название группы  
- description — описание  

---

### TransactionLink
- type — тип связи (compensation / transfer)  
- transactions — связанные транзакции  

---

## 7. Технологии

- Backend: .NET  
- Database: PostgreSQL  

Обоснование выбора PostgreSQL:
- хорошо подходит для структурированных данных  
- поддерживает сложные запросы и агрегации  
- удобен для аналитики  
- широко используется
